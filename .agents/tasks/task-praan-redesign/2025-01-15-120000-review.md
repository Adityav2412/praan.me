# Warm editorial redesign of praan.me homepage

A comprehensive visual and structural redesign that replaces the navy/cream dark aesthetic with a warm beige/sage palette, swaps the font system from Poppins+Montserrat to Playfair Display+DM Sans, rewrites the homepage composition from a dashboard-like layout to an editorial storytelling flow (Hero, LiveStatsBar, Story blocks, How It Works, Saviours Wall, Footer CTA), and creates a new `/founder` route while converting the old `/founders-message` to a client-side redirect.

Import paths resolve correctly, component props are compatible across all call sites, the existing modal flow (FormModal + CertificateModal) is preserved, and backward-compatible CSS aliases keep legacy class names functional for pages not yet touched.

Watch for: LiveStatsBar displays the same animated number for both "Live Saviours" and "Bowls placed" — they read identically at runtime (confirmed). The Footer links to a `/privacy` route that does not exist (confirmed). The `dateStyle` prop on SaviourCard is declared but never consumed in the component body (confirmed).

## High-level view

The design system foundation uses new CSS custom properties in `:root`, a `@theme inline` block mapping them to Tailwind utilities, and backward-compatible aliases so `bg-cream`, `text-navy`, `bg-navy-light` still resolve. The font swap uses `next/font/google` with proper CSS variable exposure.

The homepage rewrite replaces the old multi-widget layout with a linear editorial flow. Data-fetching logic (Google Sheets polling, localStorage caching) is preserved intact. The Navbar CTA communicates with page state through a `CustomEvent('open-form-modal')` pattern — a pragmatic workaround, but one that creates an implicit untyped contract between the navbar and page components.

The `/founder` route is a lift-and-restyle of the old `/founders-message` content. The old route becomes a client-side redirect using `router.replace`, which means search engines won't see an HTTP-level redirect for existing backlinks.

## LiveStatsBar shows identical values for two different metrics

In `components/live-stats-bar.tsx`, both the "Live Saviours" cell and the "Bowls placed" cell render the same expression:

```tsx
{hasMounted ? animatedCount : (saviourCount ?? 0)}
```

The component receives only `saviourCount` as a prop and has no separate data source for bowls placed. At runtime both stat cards display the same number, which looks like a rendering bug to users. Either distinguish the values (e.g., pass a separate `bowlsCount` prop, or apply a multiplier) or collapse to two stats instead of three.

## Dead link to /privacy in Footer

The footer includes `<Link href="/privacy">Privacy</Link>`, but no `app/privacy/page.tsx` exists. Clicking this will produce a Next.js 404. This link is newly introduced in this diff.

## SaviourCard dateStyle prop is a dead parameter

The rewritten `saviour-card.tsx` declares `dateStyle?: 'relative' | 'calendar'` and destructures it, but never references it in the JSX. The new card renders only name, colony, and saviour number — no date. `saviours-leaderboard.tsx` still passes `dateStyle="calendar"` which compiles but has no visible effect.

## CustomEvent coupling between Navbar and Page

The Navbar CTA dispatches `new CustomEvent('open-form-modal')` and `app/page.tsx` listens with `window.addEventListener('open-form-modal', handler)`. A typo in either string would silently break the "Place a bowl" button without any TypeScript error. On non-homepage routes (`/founder`, `/saviours`), the event fires but no listener exists — the navbar button does nothing on those pages. Those routes work around it by navigating to `/` for the saviour flow, but the navbar CTA itself is a no-op there.

## Client-side redirect vs server redirect for /founders-message

The old `/founders-message` URL renders a client component that calls `router.replace('/founder')` in a `useEffect`. This requires JavaScript to redirect. Search engines may not follow it as a permanent redirect, so backlinks could lose SEO value. A `redirects` entry in `next.config.mjs` would provide a proper 308 at the HTTP level without needing a client component.

## Scroll offset mismatch

The navbar height was reduced from 80px to 64px, and `NAVBAR_HEIGHT_PX` in `lib/navigation.ts` is updated to match. However, `globals.css` still declares `section[id] { scroll-margin-top: 5rem }` which is 80px at default font size. The JS-based `handleNavigate` in `page.tsx` uses the correct 64px offset, but CSS-native anchor navigation (direct URL with hash, or `<Link href="#about">`) uses the CSS `scroll-margin-top` value and will overshoot by ~16px. Update to `4rem` to match the new nav height.
